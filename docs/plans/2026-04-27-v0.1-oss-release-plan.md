**Source:** Audit on 2026-04-27 against Base UI parity, OSS norms, and the original foundation plan.

## Path: Credible OSS (~15h)

Everything below is a gate for v0.1.0. No shipping until done.

### P0 — Blockers

- [x] **Release automation via changesets.** `.changeset/` initialized, `.github/workflows/release.yml` added, root scripts wired (`pnpm changeset`, `pnpm version-packages`, `pnpm release`). Initial v0.1.0 changeset seeded.
- [x] **CHANGELOG.md.** Root changelog seeded with v0.1.0 entry; per-package changelogs will be auto-generated by changesets on each release.
- [ ] **CLI end-to-end verification.** Run `pnpm pack` on `@basex-ui/cli`, `npx ./basex-ui-cli-0.1.0.tgz add button` in a fresh Vite app. Confirm files copy, imports resolve, component renders. Add integration test at `packages/cli/src/__tests__/integration.test.ts`. If broken, fix before release.

> ~~**OTP Field component.**~~ **Removed 2026-04-27** — verified against the installed `@base-ui/react@1.2.0` package: Base UI does **not** ship an OTP Field. The original audit hallucinated this. Real Base UI 1.2.0 roster is 36 components; BaseX ships 35. The only legitimate parity gap is **Context Menu**, which is intentionally deferred to v0.2 per `docs/plans/component-roadmap.md`. v0.1 ships at 35/36 (97%) of Base UI 1.2.0.

Use `packages/components/src/toggle-group/toggle-group.test.tsx` as the floor, not the ceiling.
- [ ] **Backfill render/interaction depth on the existing 22 shallow tests.** Same bar as above. Current tests only verify `displayName` and exports.
- [ ] **CODE_OF_CONDUCT.md** — Contributor Covenant v2.1.
- [ ] **SECURITY.md** — vulnerability disclosure policy, contact email.
- [ ] **GitHub templates** — `.github/ISSUE_TEMPLATE/{bug.md,feature.md}`, `.github/pull_request_template.md`.
- [ ] **Per-package READMEs** for `tokens`, `styles`, `intelligence`, `mcp-server`, `cli`. Each: what it is, install, minimal usage example, link back to main README.
- [ ] **package.json keywords** — populate across all 6 packages: `["ui","component-library","base-ui","react","accessible","typescript"]` plus per-package specifics (cli: `"scaffold"`; mcp-server: `"mcp","ai-agents"`).

All P0 + P1 boxes ticked. `pnpm build && pnpm test && pnpm lint && pnpm typecheck` green. CI publish workflow dry-run successful. Tag `v0.1.0`.

## Post-release housekeeping (v0.2.1 shipped 2026-04-27)

The first release went out as **v0.2.1** (initial bump landed as 0.2.0 due to a `minor` changeset, then a partial publish blocked by npm scope creation forced a 0.2.1 republish — see `CHANGELOG.md`). All 6 packages are live on npm. Open items:

- [ ] **Deprecate the orphan `@basex-ui/mcp-server@0.2.0`** on npm. That version published in isolation during the failed first run; the others never made it. On npm package page → Settings → Deprecate version 0.2.0 with message: `Skipped — use 0.2.1+ for full release set`. Do **not** unpublish (npm 24h rule + breaks anyone who happened to install it).
- [ ] **Set up Trusted Publishing** per-package: npm package settings → Trusted Publishers → add `dowdavid/BaseX-UI` + `release.yml` + `refs/heads/main`. Once configured for all 6 packages, remove `NPM_TOKEN` from GitHub Secrets entirely. Trusted Publishing uses OIDC, no long-lived tokens.
- [ ] **Rotate npm token + GitHub PAT.** During the v0.1 ship, an npm token and a GitHub PAT were briefly visible in chat transcripts. The npm token has bypass-2FA, so rotate sooner rather than later (regenerate at npmjs.com → settings → tokens, update GitHub secret). The GitHub PAT (`github_pat_11A4SOE…`) was already revoked.

## Quality follow-ups (next sprint)

Owner-noted issues to address before promoting to a "stable" v1 narrative:

- [ ] **Accessibility audit + fixes.** Run axe-core / lighthouse a11y across the docs site and across each component demo. Fix flagged issues. Specific areas to check: focus order in compound components (Dialog, Drawer, Combobox), `aria-*` attribute coverage on every interactive element, screen-reader announcements for Toast and live regions, keyboard reachability of all controls, color contrast on muted/disabled states. Document the test run + fixes in `docs/plans/`.
- [ ] **Scrollbar consistency.** Standardize scrollbar styling across all scroll containers — currently inconsistent between `ScrollArea`, native overflow scrollers in demos, and the docs sidebar. Pick a single rendered scrollbar treatment (overlay, thin, color-token-driven) and apply via shared utility in `@basex-ui/styles` so it's themable and not duplicated per component.
- [ ] **Document accent color usage.** Currently `colorPrimary` (the orange) is referenced inconsistently — Button uses it for solid variant, Switch/Radio/Checkbox stopped using it (now use `colorText` post the inversion fix). Need a written rule in `DESIGN.md` or a new `docs/accent-color.md` saying *when* a component should use the brand accent vs. theme-inverting tokens vs. neutral. Then audit every component against the rule and fix mismatches.
- [ ] **Fix Toolbar width.** Toolbar root currently fills available width when it should be content-sized by default with an opt-in `fullWidth` prop (or `flex: 1` for explicit fill). Review `packages/components/src/toolbar/toolbar.tsx` styles and the demo page; align with the Base UI Toolbar API.

## Beyond v0.1.0 — Milestone Backlog

To exceed this release and earn a v0.2 / v1.0:

### v0.2 — Completeness

- [ ] **Context Menu component.** Full Base UI compound API (Trigger, Portal, Positioner, Popup, Item, Separator, RadioGroup, CheckboxItem, SubmenuTrigger, Arrow). Currently deferred in roadmap as "low priority" — that's the gap.
- [ ] **Docs bundle size <500KB gzipped.** Currently 1.3MB raw / 372KB gzipped, dominated by Shiki bundling every grammar. Lazy-load grammars and themes on demand, or swap to Starry Night.
- [ ] **Disabled-state token consistency.** Audit every component using `opacity: 0.64` (Accordion, Dialog, Combobox, others) and migrate to muted-token colors per Button's pattern. WCAG AAA-compatible disabled states across the board.
- [ ] **Accessibility test suite.** Systematic axe-core / @testing-library/jest-dom a11y assertions per component. Cover live regions (Toast), form integration (Field + Form), combobox listbox roles, focus management on Dialog/Drawer/Popover.

### v0.3 — Confidence

- [ ] **Visual regression baseline.** Chromatic or Percy in CI. Snapshot every component in every variant/state. Block PRs on unintended visual diffs.
- [ ] **Example integrations.** `examples/nextjs/`, `examples/remix/`, `examples/vite-react/`. Each is a deployable minimal app demonstrating install → use → theme.
- [ ] **Animation guide.** Doc page connecting `@basex-ui/intelligence` animation presets to component usage. "Use this preset for that interaction" with live demos.

### v1.0 — Polish & adoption

- [ ] **Type-level docs.** TSDoc on every public prop so hover hints in editors are useful.
- [ ] **Migration guides.** From shadcn/ui, from Radix, from Headless UI.
- [ ] **Theme studio.** Interactive token editor in the docs site that exports a custom theme file.
- [ ] **AI agent quickstart.** End-to-end walkthrough of using the MCP server from Claude / Cursor to scaffold a UI from intent.
- [ ] **Performance budget.** Per-component bundle size tracked in CI, regression alerts.
