# BaseX UI v0.1 — Ship Learnings

## What shipped

36 accessible React components on Base UI + StyleX, MCP server, CLI, docs site, full OSS infrastructure — public on npm as `@basex-ui/*`.

## What worked

**The compound component pattern held at scale.** Choosing it early meant every component felt consistent by the time #36 landed. No drift. The StyleX `className` callback pattern for state styling was the right call — it made all 36 components structurally identical and easy for agents to read.

**DESIGN.md as source of truth.** Enforcing token-only values caught real issues: the opacity-disabled pattern (14 components) and the accent color audit were both caught by the design system constraint, not by review. Without DESIGN.md the drift would have been invisible.

**Planning docs in `docs/plans/` per design decision.** Having per-component design docs (accordion, alert-dialog, autocomplete) meant each component started with intent, not just implementation. They became the evaluation target.

**OSS infrastructure front-loaded.** LICENSE, CODE_OF_CONDUCT, SECURITY, GitHub templates — having these before v0.1 meant the release felt credible, not scrappy. Worth the overhead.

**The `@basex-ui/intelligence` layer.** Intent resolution + anti-patterns + animation presets is genuinely differentiating. No other component library ships this. It's the thing that makes the MCP server useful vs. just a component list.

## Surprises

**The OTP Field hallucination.** Audit generated a P0 blocker for a component that doesn't exist in Base UI 1.2.0. Cost real time to verify and remove. Source: model knowledge of Base UI was stale. Fix: always verify against the installed package, not model knowledge.

**npm first publish failure.** The 0.2.0 orphan publish happened because the npm scope wasn't created before the CI release ran. One package (`mcp-server`) published in isolation; the others failed. Led to a 0.2.1 republish and a permanently orphaned 0.2.0 on npm. Fix: verify npm scope exists and all packages are claimable before wiring release CI.

**Token secret exposure in chat.** An npm token (bypass-2FA) and a GitHub PAT appeared in transcript during the ship sequence. Both were caught and rotated, but this shouldn't have happened. Fix: never paste tokens into chat — use `op run` or env injection. Add to project CLAUDE.md for any project with CI publish.

**v0.1 rated ~8.5 at ship, not 9+.** The quality pass on 2026-04-28 moved it, but several P2-P4 items (CLI expansion, visual regression, theme studio, adoption) remain open. The gap between "ships correctly" and "9+ library" is mostly non-code work: adoption signal, migration guides, AI agent quickstart.

## What to change next time

**Add a pre-publish CI checklist to the release plan template.** Should include: scope exists on npm, Trusted Publishing configured, no tokens in environment, dry-run passed. The 0.2.0 orphan and token exposure were both preventable with a checklist.

**Run the Compound phase immediately after ship, same session.** This doc is a day late. The insights are still fresh but some specifics required re-reading git log. Write learnings while the ship is still warm.
